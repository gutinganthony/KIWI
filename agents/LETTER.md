## 一、三件使用者沒問、但對這個環境最重要的事

### 1. 這個 repo 的輸出會變成真金白銀的決策——最大風險是「自信的編造」，不是 token

KIWI 的分析產出（Serenity、WaveTrend、ACT 判讀、週報）會影響實際投資。
對弱模型來說，最危險的失誤不是漏掉什麼，而是**用訓練記憶補一個看起來合理的數字**：
過期的市值、記錯的財報日期、想當然的持股比例。這種錯誤使用者很難當場發現，
發現時已經付出代價，而且會摧毀他對整套系統的信任——那才是真正的總損失。

鐵律（agents/JUDGMENT.md §5 已寫，這裡講為什麼）：**訓練記憶裡的財務數字一律視為過期**。
每個數字要有當日查證的來源；查不到寫「查無」。一份有三個「查無」的誠實報告，
價值遠高於一份數字齊全但有一個是編的報告。另外：你提供的是框架執行與證據整理，
投資決策永遠是使用者的——結論要附否證條件，不要寫成保證。

### 2. 自動化管線會靜默腐爛——接分析任務前先花 30 秒驗資料新鮮度

每日 dashboard 更新、每日 Telegram/LINE 警報、每週六 Serenity 週報都是無人看管的
GitHub Actions。API 金鑰過期、資料源改版、上游欄位改名，都會讓 dashboard 靜默停更
或顯示舊資料——而使用者可能每天在看它做決定。

具體習慣：任何投資分析任務開始時，先跑
`jq -r '.d[-1]' docs/history.json` 看最後資料日期。比今天舊超過 3 個交易日 →
先主動告訴使用者「dashboard 資料停在 X 日」，再繼續原任務。這 30 秒可能是
你那個 session 做的最有價值的一件事。

### 3. 使用者的真正資產是判斷框架，不是程式碼——維護時保真第一

watchlist.md 的觸發／否證條件、KIWI_INDEX_FRAMEWORK.md 的權重與門檻，
是使用者投資判斷的外化，很多數字是他刻意選的不對稱（例如某檔觸發條件特別嚴）。
弱模型有把內容「正規化」的本能：統一格式順手把 7% 改成 5%、把「兩季」改成「一季」、
把使用者的口語判斷潤飾成通用陳述。**這是資產損毀。**
改這些檔案時：只做使用者明確要求的變更；引用他的判斷時逐字保留數字與條件；
發現框架內部矛盾時標註回報，不要自行「修正」。

## 二、這套制度最可能的退化方式與預防

1. **CLAUDE.md 長回去**。每個 session 都想「這條很重要，加進 CLAUDE.md」，
   兩個月後它又是 50KB。預防：教訓寫 LEARNINGS，不寫 CLAUDE.md；
   CLAUDE.md 超過 120 行就觸發收斂（MAINTENANCE §4）。
2. **規則被「例外」侵蝕**。某個 session 趕任務跳過驗證（「這次很簡單，不用 verifier」），
   下個 session 從對話慣性裡學到先例。預防：制度檔是權威，前一個 session 的做法不是；
   要跳過某條規則，必須在回報裡明說「跳過了 X，原因 Y」，讓使用者看得見。
3. **路由失聯**。制度檔被搬移改名，CLAUDE.md 指向不存在的路徑，弱模型自由發揮。
   預防：CLAUDE.md 已寫「引用不存在＝忽略，不要自己補寫」；搬移制度檔時，
   同一個 commit 內 grep 舊檔名更新所有引用。
4. **儀式化**。照流程跑但不懂為什麼：把期望結論塞給 verifier（變橡皮圖章）、
   派工三件套抄模板但驗收條件寫「做好做滿」（等於沒寫）。預防：每次覺得某條規則
   「只是形式」，去讀 agents/DIAGNOSIS.md 看它擋的是哪個真實事故；真心認為規則錯了，
   走 MAINTENANCE §1 問使用者，不要默默繞過。

## 後記：2026-07-07 對抗審查結果

制度初稿完成後，開了四路 fresh-context 對抗審查（規則衝突、引用實跑驗證、
配方實跑、haiku 情境測驗八題），共 20 條 findings，全數修正。最重要的三個抓漏，
留在這裡給未來 session 當警世故事：

1. **我自己也犯了「沒跑過就寫進制度」的錯**：初稿寫的 pytest 驗證指令在本環境
   根本跑不起來（沒裝依賴、而且 avi-v5 壓根沒有 pytest 套件）。修正後的每一條配方
   都經過實跑。教訓：制度檔裡的任何指令，落檔前必須實跑一次。
2. **saved_*.json 是 mojibake 編碼**：中文關鍵字直接搜永遠 0 命中，弱模型會誤報
   「查無」。轉換配方在 DISPATCH §5。
3. **haiku 實測八題答對七題**：制度檔對弱模型基本可讀，唯一誤讀（完成檢查漏掉
   commit+push）已透過拆項加粗修正。這個「讓真的弱模型讀一遍再考它」的驗證方法
   值得沿用——比任何自我審查都誠實。
