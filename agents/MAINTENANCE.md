# 制度檔案維護協議（交付項 F）

> 讀者：未來想要修改 CLAUDE.md 或 agents/ 下任何檔案的模型。
> 原則：制度檔是所有 session 的共用基礎設施，改壞一次、之後每個 session 都跟著壞。

## §1 修改權限表

**可以自行改（不用問使用者）**：

| 檔案／範圍 | 允許的操作 | 條件 |
|---|---|---|
| agents/LEARNINGS.md | 依 §3 格式新增或更新條目；依 §4 精簡 | 踩到新坑或坑已固化 |
| agents/DISPATCH.md §5 | 更新大檔案配方 | 檔案大小/結構實際變了，且新配方你實跑驗證過 |
| agents/TEMPLATES.md | 新增模板或補充範例 | 不刪不改既有模板骨架 |
| CLAUDE.md §2 黑名單 | 新增條目 | 發現新的 >50KB 常駐檔案 |
| CLAUDE.md §3 路由表 | 新增一列 | 使用者新增了分析模組或新任務型態 |

**先問使用者才能改**：

- 刪除或放寬任何既有規則（門檻調鬆、黑名單移除、驗證步驟省略）。
- agents/JUDGMENT.md 的任何判準。
- CLAUDE.md 的結構、§0 環境事實、§1 gstack 段落。
- `.claude/agents/*.md`（verifier/applier 的 frontmatter 與正文）。
- `.claude/settings.json`、`.claude/hooks/`、`.github/workflows/`。

**永遠不改（唯讀歷史文件）**：

- `agents/backups/` 內所有檔案。
- `agents/DIAGNOSIS.md`、`agents/LETTER.md` —— 只允許在文末追加「後記：[日期] …」，不允許改動原文。

## §2 修改流程（每次都一樣）

1. 備份：`cp 檔名 agents/backups/檔名.$(date +%F).bak`（同一天第二次改同檔可略）。
2. 修改。
3. read-back：用 Read 把改動段落讀回來，逐字核對。
4. 自問一個具體場景：「Haiku 等級的模型讀到這段，會不會做錯？」有疑慮就改到沒有歧義。
5. commit（訊息用 `docs(agents): ...` 前綴）＋ push **到目前工作分支**，確認 push 輸出成功。
   若目前在 main：LEARNINGS 條目與配方類事實校正可直接 push（視同既有授權）；
   其他制度檔改動先開分支，或按 agents/JUDGMENT.md §3 問使用者。

## §3 教訓回寫格式（agents/LEARNINGS.md）

一條一行起、最多 4 行，固定欄位：

```
- [YYYY-MM-DD] 情境：發生什麼事｜錯誤：做錯了什麼（或：事實：發現了什麼）｜正解：正確做法｜規則：一句未來可直接執行的話
```

寫入前先檢查是否已有同類條目——**有就更新那一條**（改日期、修正內容），不要重複新增。
值得寫的：會再發生的環境事實、工具行為、資料格式陷阱。
不值得寫的：一次性的任務細節、已寫進制度檔的內容、與本 repo 無關的通用知識。

## §4 精簡門檻

`agents/LEARNINGS.md` 超過 **60 行或 15 條** 時，執行一次精簡（可自行做）：

1. 備份（§2 流程）。
2. 已固化的規則搬家：路由類 → CLAUDE.md §3；派工/工具類 → DISPATCH.md；判斷類 → JUDGMENT.md（注意：JUDGMENT 判準要先問使用者才能改，搬過去前先問）。
3. 過時條目（環境已變、檔案已刪）直接刪除。
4. 剩餘條目合併同類、壓縮字數。
5. 精簡後在檔案開頭更新「上次精簡：日期」。

CLAUDE.md 有自己的膨脹門檻：**超過 120 行**就該問使用者要不要做一輪收斂（把細節下放到 agents/ 檔案）。CLAUDE.md 每一行都是每個 session 的常駐成本。

## §5 環境事實過期時

DISPATCH.md §2 的模型清單、工具行為，以**當下 session 實際的工具 schema 為準**。
發現不一致（例如 model 參數多了/少了選項、某 MCP 工具消失）：
1. 以實際 schema 行動（不要照舊文件硬呼叫）。
2. 把差異記進 LEARNINGS.md。
3. DISPATCH.md §2 的更新屬於「先問使用者」等級？——**不是**：純事實校正（清單與 schema 對齊）可自行改並註記日期；改「指派邏輯」（哪類任務用哪個模型）才要先問。
