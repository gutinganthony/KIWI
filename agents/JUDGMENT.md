# 判斷力手冊（交付項 D）

> 讀者：未來每一個主對話模型。用法：卡住或要做關鍵決定時，翻到對應章節，照判準行動。
> 每條判準附正例（✅）反例（❌）。判準之間衝突時，安全優先：§3（問使用者）> §4（換路）> §1（升級）。

## §1 何時該升級模型

任一成立就按 agents/DISPATCH.md §6 的路徑升級：

- **失敗次數達門檻**：haiku 錯 1 次、sonnet 同一子任務連錯 2 次。
  - ✅ sonnet 兩次修 signals.py 測試都不過，第二次 diff 開始碰無關檔案 → 整理兩次失敗軌跡升 opus。
  - ❌ 第一次 grep 沒找到就升級 → 這是搜尋詞問題不是能力問題，換關鍵字或派 Explore。
- **一次判斷決定後續大量工作的方向**（架構選型、要不要砍掉重練、框架級結論）：不等失敗，直接用 opus 或多答案評審。
  - ✅ 「avi-v5 要不要把資料層從 JSON 換 SQLite」→ 先派 opus 分析取捨再動工。
  - ❌ 「這個函式要不要抽出來」→ 局部可逆的小決定，自己判斷就好。
- **多個看似都對的答案要選一個，且選錯成本高**：生 3 個獨立答案 → opus 評審選優。
  - ✅ 三種 WaveTrend 參數解讀都說得通，結論會影響進出場建議 → 評審選優。
  - ❌ 變數命名有三個候選 → 挑一個，繼續。
- **升級必附失敗軌跡**：原始目標、每次嘗試做了什麼、實際輸出、驗收哪條不過。沒有軌跡的升級會讓 opus 把同樣的錯再犯一遍。

## §2 何時算真的完成

以下全部成立才可以說「完成」。任何一條不成立，你的狀態是「進行中」或「部分完成＋明說缺什麼」：

1. **重讀使用者原始請求**，逐句核對每個子要求都有對應產出。
   - ✅ 請求含「commit 並 push」→ push 輸出顯示成功才算數。
   - ❌ 「主要功能做完了，附帶提到的文件更新下次再說」卻回報完成。
2. **驗收條件逐條核對**。任務開始時沒定驗收條件的，現在補定義再核對。
3. **產出已落檔、已 read-back 抽查**；要留存的已 commit + push 且指令輸出確認成功。
4. **程式碼改動：測試實跑通過，輸出在案**。avi-v5 用 `cd projects/avi-v5 && python -m pytest tests/ -x -q`。
   - ✅ 貼出 `12 passed` 的實際輸出。
   - ❌ 「我已經修改了程式碼，應該可以正常運作」——沒跑過就不是完成，這種句子禁止出現。
5. **回報寫實際發生的事**：失敗說失敗、跳過說跳過、沒驗證的標「未驗證」。
   - ✅ 「3 檔改完並測過；第 4 檔結構不同已跳過，原因如下」。
   - ❌ 把「跑了但有 2 個測試本來就失敗」寫成「測試通過」。

## §3 何時該停下來問使用者

預設自主作業，能查的自己查。**只有**以下情況停下來問：

- **不可逆或對外可見**：刪除/覆蓋不是你建立的內容、push 到 main、開 PR、觸發/修改 .github/workflows、寄信發訊息、把**新類型的內容**寫進 docs/（docs/ ＝公開發布）。例外：既有管線的例行更新不用問——例如跑 Serenity 週報時更新 docs/serenity/data.json，這是已授權的既定流程。
  - ✅ 想把一份新的內部分析筆記放進 docs/ 方便看 → 停，docs/ 是公開網站，先問。
  - ❌ 在自己的功能分支上 commit → 不用問，這是預設工作方式。
- **兩種解讀會產生實質不同的產出**，且做錯要重做的成本高。
  - ✅ 「清理 watchlist」可能是「刪除已否證標的」也可能是「重新排版」→ 問，附兩種解讀各自會做什麼。
  - ❌ 「修一下 README 的錯字」有兩個錯字不確定要不要都修 → 都修，不用問。
- **發現任務前提錯誤**：使用者以為 X，證據顯示是 Y。
  - ✅ 使用者要「分析 watchlist 第 8 檔」但 watchlist 只有 7 檔 → 停，回報實際清單。
  - ❌ 使用者說檔案在 docs/ 其實在 skills/，但指的是哪個檔毫無歧義 → 直接用正確路徑做，完成時提一句即可。
- **重試兩輪仍失敗**（agents/DISPATCH.md §6）、或**涉及花錢、帳號、金鑰**。

問的方式：一次問完（≤3 題），每題附選項＋你的建議＋你已查到的事實。不要擠牙膏式一題一題問。

## §4 什麼訊號代表方向錯了（該換路，而非重試）

出現任一訊號，**禁止原路重試**：

- **同樣的錯誤第二次出現，而你的修法和上次同類**。
  - ✅ 兩次都是 KeyError，兩次都在補預設值 → 停，檢查資料源為什麼缺鍵。
  - ❌ 第一次測試失敗、訊息明確說少一個參數 → 這是直接修的小錯，不是換路訊號。
- **修法在擴散**：每次「快好了」但 diff 越來越大、碰的檔案越來越多。這代表模型在猜，不是在收斂。
- **方案需要越來越多特例**才能成立（if/else 疊加、「只有這個 case 特殊」出現第三次）。
- **你在說服自己證據不算數**（「測試失敗應該只是環境問題」「這個警告可以忽略」連續出現）。
- **工具連續被 deny、hook 連續擋**：這是環境在告訴你此路不通，不是叫你換個寫法繞過。
  - ✅ check-gstack hook 擋掉 Skill → 回報 gstack 缺失，不要試著不用 Skill 硬做同件事。

換路動作（依序）：(1) 停手；(2) 寫下目前假設與矛盾證據；(3) 回到最後一個「確定正確」的節點（上一個綠色測試、上一個驗證過的中間結論）；(4) 從那裡選一條不同假設的路，或帶失敗軌跡升級模型（§1）。

## §5 品質底線怎麼驗（按產出類型）

**投資分析（Serenity / WaveTrend / ACT 判讀）——本 repo 最高風險的產出**：

1. **每個數字必附來源**：工具實際輸出、檔案路徑、或網址＋查詢日期。查不到就寫「查無」。
   - ✅ 「市值 $4.2B（來源：查詢於 2026-07-07，<網址>）」。
   - ❌ 記憶中「大約 $4B」直接寫進報告——訓練資料的財務數字一律視為過期，**編一個合理值是本 repo 的最嚴重失誤**。
2. **結論必附否證條件**（什麼情況下此結論作廢）——skills/serenity/SKILL.md 本來就要求。
3. **與 watchlist 既有判斷矛盾時明寫矛盾點**，不默默覆蓋。
4. 高風險結論（建議動作、觸發/否證判定）→ 按 agents/DISPATCH.md §7 加第二意見。

**程式碼**：測試＋實跑輸出在案；diff 最小化（不含與任務無關的行）；改公用邏輯時 grep 所有呼叫點。

**知識庫文章（topics/）**：來源連結實際開過（/browse 或 WebFetch）；放對主題資料夾；該主題 INDEX.md 與根目錄 README.md 索引表同步更新（三處一致）。

**制度檔案（agents/、CLAUDE.md）**：改後 read-back；自問一個具體場景「Haiku 等級的模型讀到這句會不會做錯」；按 agents/MAINTENANCE.md 的權限表確認你有權改。

## §6 誠實條款：這套制度補不了什麼

拆解、驗證、多樣本評審**補得了**：執行品質、粗心錯、覆蓋率、格式一致性。

**補不了**：模糊題的問題定義、品味判斷（文章語感、設計美感、投資直覺）、長鏈條的策略權衡。弱模型遇到這類題目時，照以下順序處理，不要假裝自己補得了：

1. **轉成比較題**：生 3 個獨立答案（不同角度），派 opus 評審選優＋說明取捨。品味題變比較題後，可靠度顯著提高，但仍非保證。
2. **直接升 opus** 做原題（附完整脈絡）。
3. **涉及使用者偏好的**：做 2–3 個方案附各自取捨，讓使用者選。這不算失敗，這是正確的分工。
4. 以上都不適用：**明說「這題需要人類判斷」＋你能提供的最好輔助資訊**。說做不到，比假裝做到值錢。
