# 派工 Prompt 模板（交付項 E）

> 用法：選對應任務型態的模板 → 把 `【】` 填空填完 → 整段貼進 Agent 工具的 prompt 參數。
> `subagent_type` 與 `model` 按模板開頭的建議填；不確定就省略 model（＝繼承主對話模型）。
> 三件套（目標與動機／驗收條件／回報格式）是模板骨架，**填空可以改，骨架不准刪**。
> 派工後收到超出格式的回報：摘要要點後丟棄原文，不要整段留在對話裡。

---

## T1 搜尋／定位（subagent_type: Explore，model: 省略）

```
目標：找出【要找什麼：檔案/函式/設定/文字】的確切位置。
動機:【為什麼要找，找到之後要拿來做什麼——這決定回報詳細度】。
範圍：【目錄或 glob，例 projects/avi-v5/**/*.py；不確定就寫「整個 repo，但跳過
saved_collections.json、saved_posts.json、docs/index.html、docs/history.json」】。
搜尋深度：【medium ／ very thorough（多命名慣例、多位置）】。

驗收條件：
- 每個結果附 檔案:行號。
- 找不到也是有效答案，但必須列出你試過的搜尋詞與看過的目錄，證明找過。

回報格式（≤15 行）：
結論：一句話。
位置清單：檔案:行號 — 各一行說明。
沒找到的話：試過的搜尋詞／看過的位置。
```

## T2 實作（subagent_type: general-purpose，model: 省略；已打樣的批次件用 applier）

> **若這是修 bug**：把驗收條件第一條改成「先加一條能重現該 bug 的失敗測試/斷言，貼出它『先紅、修後轉綠』的實跑輸出」——沒先重現就修＝沒證據證明修對了東西（JUDGMENT §5 程式碼段）。

```
目標：【要做出什麼，一句話】。
動機：【使用者原始需求的一句話摘要＋這個子任務在整體中的角色】。
現狀與脈絡：【相關檔案路徑＋已知的關鍵事實，例：「訊號邏輯在
projects/avi-v5/backtest/signals.py，權重定義在 projects/avi-v5/config/avi_weights.yaml」。
路徑一律寫 repo-root 完整路徑。不要叫 agent 自己重新探索你已經知道的事】。
邊界：只改【檔案清單】；不碰【明確排除項，例 .github/workflows/、docs/】。

驗收條件：
- 【具體行為改變，例：「實跑 cd projects/avi-v5 && python3 <改動的腳本> 成功且輸出含新欄位」】
- 驗證通過：【驗證指令；avi-v5 沒有 pytest 套件，正確配方見 agents/DISPATCH.md §7】，
  以實際執行輸出為準。
- diff 不含與目標無關的行。

回報格式（≤15 行）：
結論：完成／部分完成／失敗。
改動：檔案:行號範圍 — 各一行說明。
測試輸出：最後 3 行原文。
未處理事項與原因（沒有就寫「無」）。
```

## T3 重構（subagent_type: general-purpose，model: 省略；跨模組大重構先派 Plan）

```
目標：把【對象】從【現狀】重構為【目標形態】，外部行為不變。
動機：【為什麼值得重構——這決定取捨方向】。
不變式（重構的鐵律）：
- 【列出不能變的可觀察行為，例：「所有現有測試不改動且全過」「CLI 參數不變」
  「輸出 JSON schema 不變」】。

驗收條件：
- 重構前先跑一次測試/驗證指令記錄基線（avi-v5 的驗證配方見 agents/DISPATCH.md §7），
  重構後同指令結果不劣於基線（貼兩次輸出）。
- 【若無測試覆蓋：先為現行為寫最小特徵測試（golden test），再動手改】。
- 每個被移動/改名的符號，grep 確認無殘留舊引用（貼 grep 指令與結果）。

回報格式（≤15 行）：
結論＋基線對比（改前/改後測試輸出各 1 行）。
結構變化：舊位置 → 新位置 清單。
風險註記：你不確定的點（沒有就寫「無」）。
```

## T4 研究（subagent_type: general-purpose，model: 省略；結論影響投資判斷時升 opus 或加第二意見）

```
研究問題：【一句話，可回答的形式，不要開放式「了解一下」】。
動機：【答案會被拿去做什麼決定】。
工具白名單（研究 session repo scope 外的 GitHub repo 時**必寫**）：只准 WebSearch／WebFetch，
禁用 mcp__github__* 搜尋類工具，避免 subagent 跨 scope 查詢。
來源要求：優先【官方文件/財報/一手來源】；每個關鍵事實附 網址＋查詢日期；
社群貼文與轉述當線索不當證據。網頁瀏覽用 /browse 或 WebFetch。
截止思考：查不到的就標「查無」，禁止用訓練記憶補財務數字或日期。

驗收條件：
- 回答直接回應研究問題（不是丟一堆相關資訊）。
- 每個數字與關鍵主張有來源標註。
- 明確區分：事實（有來源）／推論（你的分析）／不確定（查無或來源互相矛盾）。

回報格式：
結論：≤3 行直接回答。
證據表：主張 — 來源網址 — 日期（≤8 條，挑最關鍵的）。
反面訊號：與結論矛盾的發現（沒有就寫「無」——但先真的找過）。
長版報告（若有）：寫到【檔案路徑】，這裡只給路徑。
```

## T5 審查（subagent_type: verifier；verifier 不可用時 general-purpose + model: sonnet）

```
受驗產出：【檔案路徑或 diff 範圍】。
產出目標的一句話描述：【它應該做到什麼】。
唯讀約束：你的審查是唯讀的——不得改動任何檔案，也不得動 git index/HEAD/branch。
要看別的 revision 用 `git worktree add /tmp/review-<SHA>`，絕不移動 HEAD。發現問題只回報。
（注意：不要附上產出者的推理過程或你期望的結論——讓審查者獨立判斷。）

驗收條件（逐條判定 通過/不通過/無法驗證）：
1.【條件一，可核對的形式】
2.【條件二】
3.【…每條都要具體到審查者知道「怎麼查」】

回報格式（≤20 行）：
總判定：可交付？是／否（擋交付項 N 個）／修 N 項後可交付。
逐條：條件 → 判定 — 證據（檔案:行號 或 指令輸出 ≤2 行）。
問題分級（每項附 檔案:行號＋錯在哪／為何重要／怎麼修）：
  Critical（擋交付，必修）／Important（該修）／Minor（可延後）；某級別沒有就寫「無」。
未涵蓋風險：驗收條件沒問到但你注意到的問題（≤3 條）。
```

## 附：升級重派時的失敗軌跡附錄（接在任何模板後面）

```
【前情】此任務已失敗 N 次，失敗軌跡：
- 第 1 次（model:【】）：做了【】→ 結果【實際輸出/錯誤 ≤3 行】→ 驗收未過原因【】。
- 第 2 次（model:【】）：改為【】→ 結果【】→ 未過原因【】。
請先診斷前幾次共同的錯誤假設，再開始動手；若你認同其中某次的方向，說明為何它會差一步。
```
