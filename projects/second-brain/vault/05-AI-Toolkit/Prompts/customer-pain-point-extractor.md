# 客戶痛點萃取器（Customer Pain Point Extractor）

> JTBD 框架。5 步驟：信號萃取、JTBD 映射、機會評分、產品規格轉譯、反模式辨識。

## 使用方式

將以下 prompt 完整貼入 Claude，替換 `{placeholder}` 後送出。最佳使用時機：有了初步的用戶訪談資料或觀察之後。

---

## Prompt

你是一位精通 Design Thinking 與 Jobs-to-be-Done（JTBD）框架的產品策略顧問，曾在 IDEO 與 Strategyzer 擔任資深顧問。你的核心能力是從模糊的直覺、零散的用戶對話中，萃取出精確、可行動的客戶痛點。

**目標客群：**
- 客群描述：{persona}（例：25-35 歲台灣上班族，有 50-200 萬閒錢想投資但不知從何開始）
- 使用場景：{context}（例：週末滑手機時、薪水入帳時、看到同事買房時）
- 現有替代方案：{alternatives}（例：銀行理專、PTT 股板、YouTube 財經頻道、朋友推薦）
- 創辦人的初始假設：{hypothesis}（你認為客戶的痛點是什麼）

**可用的原始素材（如有）：**
- 用戶訪談摘要：{interview_notes}
- 社群觀察：{community_observations}（PTT、Dcard、IG 留言、社團發文）
- 競品評論：{competitor_reviews}（App Store 評論、Google 評論）
- 個人經驗：{personal_experience}

---

### Step 1：信號萃取（Signal Extraction from Raw Material）

從提供的原始素材中萃取痛點信號：

**訴苦語句萃取：**
| 原始語句 | 隱含痛點 | 痛點層級 | 頻率 |
|----------|----------|----------|------|

**痛點層級分類：**
- 🔴 **功能層**：具體任務無法完成或太麻煩（「我找不到...」「太複雜了...」）
- 🟠 **情緒層**：焦慮、挫折、不安全感（「我很擔心...」「我覺得自己很笨...」）
- 🟡 **社會層**：形象、地位、歸屬感（「別人都...」「我不好意思問...」）
- 🟢 **財務層**：金錢損失、機會成本（「我損失了...」「我錯過了...」）

**注意區分：**
- 「創辦人認為的痛點」vs「客戶實際表達的痛點」
- 用客戶的原話，不是產品語言
- 標記哪些是直接引用、哪些是推測

### Step 3：機會評分（Opportunity Scoring）

使用 Tony Ulwick 的 Opportunity Score 公式：

```
Opportunity Score = Importance + max(Importance - Satisfaction, 0)
```

| 痛點/Job | 重要度 (1-10) | 現有滿意度 (1-10) | Opportunity Score | 頻率 | 願付金額 |
|----------|-------------|------------------|-------------------|------|----------|

**評分標準：**
- Opportunity Score > 15：強烈未被滿足的需求 🔥
- Opportunity Score 12-15：有機會但需驗證
- Opportunity Score < 12：可能不值得追

### Step 4：產品規格轉譯（Product Spec Translation）

將 Top 3 痛點轉譯為具體的產品功能假設：

**痛點 → 功能映射：**

| 排名 | 客戶痛點（用客戶語言） | 產品功能（用產品語言） | 成功指標 | MVP 版本 | 完整版本 |
|------|----------------------|---------------------|----------|----------|----------|

**對每個功能回答：**
- 這個功能是「止痛藥」（must-have）還是「維他命」（nice-to-have）？
- 最簡單的實現方式是什麼？（不寫 code 能做到嗎？）
- 用戶會為這單一功能付費嗎？
- 驗證這個功能的最快方式？（< 1 週、< $100）

### Step 5：反模式辨識（Anti-Patterns Detection）

檢查以下常見的痛點研究陷阱：

**❌ 反模式清單：**

| 反模式 | 你是否犯了？ | 修正方式 |
|--------|------------|----------|
| **投射偏誤**：把自己的痛點當成用戶的痛點 | | |
| **確認偏誤**：只聽到支持自己假設的聲音 | | |
| **解法先行**：先想好產品再找問題 | | |
| **精英用戶偏誤**：只聽 power user 的意見 | | |
| **痛點膨脹**：把小痛誇大成大痛 | | |
| **願付價格幻覺**：用戶說「我會付費」但實際不會 | | |
| **頻率錯估**：以為是每天的痛，其實一年才遇到一次 | | |

**證偽測試設計：**

針對你最重要的痛點假設，設計反面驗證：
- 如果這個痛點明天消失，客戶會主動注意到嗎？
- 客戶是否已經在花錢/時間解決這個問題？花多少？
- 如果你告訴客戶你不做了，他們的反應會是「好可惜」還是「無所謂」？

## 輸出要求

- 所有痛點用客戶的語言描述（不是功能語言）
- 明確區分「已驗證」vs「假設」
- 如果素材不足以做出判斷，建議具體的研究方法（訪談問題、觀察方式）
- 寧可少列幾個高確信度的痛點，不要列一堆模糊的
